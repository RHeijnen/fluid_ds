`<fluid-toolbar>` groups a related set of frequently-used controls (buttons,
toggles, links) and implements the WAI-ARIA Toolbar pattern. The whole toolbar
is a single Tab stop: Tab moves into and out of it, while Arrow keys move
between the controls inside. Home and End jump to the first and last control.

<Demo>
  <fluid-toolbar aria-label="Text formatting">
    <fluid-button variant="ghost">Bold</fluid-button>
    <fluid-button variant="ghost">Italic</fluid-button>
    <fluid-button variant="ghost">Underline</fluid-button>
  </fluid-toolbar>
</Demo>

### Disabled controls

Controls that are `disabled` (or `aria-disabled="true"`) are skipped by the
roving tabindex, so Arrow keys jump straight over them.

    <fluid-button variant="ghost" disabled>Redo</fluid-button>
    <fluid-button variant="ghost">Clear</fluid-button>
  </fluid-toolbar>
</Demo>

| Key | Action |
| --- | --- |
| Tab / Shift+Tab | Move into the toolbar, then out of it (the toolbar is one Tab stop). |
| Arrow Right / Left | Move between controls in a horizontal toolbar (wraps at the ends). |
| Arrow Down / Up | Move between controls in a vertical toolbar (wraps at the ends). |
| Home | Focus the first control. |
| End | Focus the last control. |

## When to use

- A text editor's formatting controls (bold, italic, alignment).
- A media player's transport controls (play, pause, skip).
- A grouped row of actions that all operate on the same context.

Use a plain layout (a [`<fluid-button-group>`](/components/button-group/) or
flex row) when the controls are not a single composite widget, or are each their
own Tab stop. Do not nest toolbars: the pattern does not support it.

## Theming

Every styled property reads a **component-scoped `--fluid-toolbar-*` token that
falls back to a main semantic var** (the override ladder). The **complete list**
is in the [API table below](#api).

```css title="brand.css"
[data-fluid-brand="custom"] {
  --fluid-toolbar-bg: var(--fluid-surface-base);
  --fluid-toolbar-border: var(--fluid-border-default);
  --fluid-toolbar-radius: var(--fluid-radius-md);
  --fluid-toolbar-gap: var(--fluid-space-1);
}
```

### Beyond tokens: `::part()`

The container is [`part="base"`](#api). Reach it with
[`::part()`](https://developer.mozilla.org/en-US/docs/Web/CSS/::part) for CSS not
covered by a token.

## Accessibility

- The container carries `role="toolbar"` and reflects `orientation` to
  `aria-orientation`.
- Name the toolbar with `aria-label` (or `aria-labelledby`) on the host: the
  name is forwarded onto the element with `role="toolbar"`.
- Focus management uses a **roving tabindex**: exactly one control is in the Tab
  order at a time, so the toolbar is a single Tab stop and Arrow keys move
  between controls. Home and End jump to the ends.
- Disabled controls (native `disabled` or `aria-disabled="true"`) are skipped by
  Arrow navigation and never receive the tab stop.

### AA vs AAA

The toolbar itself renders no text and sets no fixed target sizes, so it has no
contrast or target-size obligations of its own: those belong to the controls you
slot in (use Fluid controls, which read `--fluid-target-min` and the focus-ring
tokens, so they scale to 44x44 targets and enhanced focus under
`data-fluid-conformance="aaa"`). The toolbar's only conformance duty, a correct
single-Tab-stop focus model, is met identically at AA and AAA.
