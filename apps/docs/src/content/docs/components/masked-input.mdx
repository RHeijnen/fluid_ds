`<fluid-masked-input>` is a form-associated text field that formats input against
a fixed-width `mask` as the user types. Mask placeholders accept input (`#` = a
digit, `A` = a letter, `*` = alphanumeric) and any other character is a literal
the field inserts automatically. The submitted form value is the formatted
string; the raw characters the user typed are exposed as the read-only
`unmaskedValue` property.

## Examples

### Masks

The `mask` string drives everything. Each placeholder token accepts one
character; literals between them are inserted for the user as they type.

| Token | Accepts |
| ----- | ------- |
| `#`   | A digit (0-9) |
| `A`   | A letter (a-z, A-Z) |
| `*`   | A letter or a digit |

### Reading the raw value

The form submits the **formatted** string. When you need the digits alone (for an
API call, say), read the `unmaskedValue` property.

<Tabs syncKey="framework">
  <TabItem label="HTML">
    ```html
    <fluid-masked-input id="phone" mask="(###) ###-####"></fluid-masked-input>

    <script type="module">
      const el = document.getElementById("phone");
      el.addEventListener("fluid-change", (e) => {
        console.log(e.detail.value);          // "(555) 123-4567"
        console.log(e.detail.unmaskedValue);  // "5551234567"
      });
    </script>
    ```
  </TabItem>

### In a form

The field participates in `<form>` submission and validation like a native
`<input>`. Mark it `required` to reject an empty or incomplete value.

## When to use

- Phone numbers, dates, card numbers, and other fixed-format entries where the
  delimiters are predictable.
- Codes with a known shape (postal codes, serials) where auto-inserting the
  separators speeds entry and prevents typos.

Reach for a plain [`<fluid-input>`](/components/input/) when the value is
free-form or its length varies. For one-time passcodes split across cells, use
[`<fluid-otp>`](/components/otp/).

## Theming

Every styled property reads a **component-scoped `--fluid-masked-input-*` token
that falls back to a main semantic var** (the override ladder), and the field
mirrors `<fluid-input>`'s field tokens (`--fluid-field-*`) so masked and plain
fields line up. The **complete list** is in the [API table below](#api).

```css title="brand.css"
[data-fluid-brand="custom"] {
  --fluid-masked-input-bg: var(--fluid-surface-base);
  --fluid-masked-input-border-focus: var(--fluid-accent-base);
  --fluid-masked-input-radius: 8px;
}
```

### Beyond tokens: `::part()`

The field shell is [`part="base"`](#api) and the inner control is `part="input"`.
Reach them with [`::part()`](https://developer.mozilla.org/en-US/docs/Web/CSS/::part)
for any CSS not covered by a token.

## Accessibility

- The field wraps a real `<input>`, so it inherits native keyboard behavior:
  typing, selection, and **Backspace skips back over inserted literals** to the
  previous placeholder.
- Provide an accessible name: set `aria-label` on the host (forwarded to the
  inner input) or associate a visible `<label>`.
- `aria-invalid` is always present on the inner input and flips to `true` when a
  `required` field is left empty or incomplete. Validity is conveyed through the
  ARIA state, not by border color alone (SC 1.4.1).
- The invalid border uses the theme-independent `--fluid-danger-base` token, so
  it keeps a 3:1 contrast against the field across every brand and in dark mode.

### AA vs AAA

The field reads the conformance tokens rather than hard-coding sizes. Under the
default **AA** mode every size variant keeps its design height (min 24 CSS px
target). When an ancestor opts into **AAA** with `data-fluid-conformance="aaa"`,
`--fluid-target-min` lifts the field to a 44×44 target (SC 2.5.5) and
`--fluid-focus-ring-width` thickens the focus ring, with no per-component
branching.
