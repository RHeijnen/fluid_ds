`<fluid-description-list>` lays out a set of key/value pairs. Each
`<fluid-description-item>` pairs a term (the key, via the `term` slot) with its
detail (the value, in the default slot). It is presentational and semantic: the
pairs form a list, so the container exposes `role="list"` with each item as
`role="listitem"`.

<Demo>
  <fluid-description-list aria-label="Account details" style="max-width: 32rem;">
    <fluid-description-item>
      <span slot="term">Name</span>
      Ada Lovelace
    </fluid-description-item>
    <fluid-description-item>
      <span slot="term">Email</span>
      ada@example.com
    </fluid-description-item>
    <fluid-description-item>
      <span slot="term">Role</span>
      Administrator
    </fluid-description-item>
  </fluid-description-list>
</Demo>

for denser panels where the rows benefit from a visible separator.

### Multiple columns

Set `columns` to a value greater than 1 to lay the pairs out as a responsive
grid. It fits up to that many equal columns when the width allows, then
collapses to fewer columns (down to one) on narrow viewports, so it never
overflows its container.

### Rich detail content

Put anything in the detail slot: text, links, or other components. Long values
wrap within their column.

<Demo>

      <span slot="term">Summary</span>
      <span>Framework-agnostic web-component design system built on Lit and TypeScript.</span>
    </fluid-description-item>
    <fluid-description-item>
      <span slot="term">Homepage</span>
      <a href="https://example.com">example.com</a>
    </fluid-description-item>
  </fluid-description-list>
</Demo>

## When to use

- Profile and account detail panels (name, email, role).
- Order or invoice summaries (number, status, total).
- Specification sheets and metadata blocks.

Reach for a [`<fluid-table>`](/components/table/) instead when you have many
records that share the same fields and benefit from columns, sorting, or
selection. A description list describes one subject.

## Theming

Every styled property reads a **component-scoped `--fluid-description-list-*` /
`--fluid-description-item-*` token that falls back to a main semantic var** (the
override ladder). Override the row gap, column gap, divider color, or the term
and detail typography per instance, per component, or per brand. The **complete
list** is in the [API tables below](#api).

```css title="brand.css"
[data-fluid-brand="custom"] {
  --fluid-description-list-gap: var(--fluid-space-4);
  --fluid-description-list-divider-color: var(--fluid-border-default);
  --fluid-description-item-term-fg: var(--fluid-text-secondary);
  --fluid-description-item-detail-fg: var(--fluid-text-primary);
}
```

### Beyond tokens: `::part()`

The list is [`part="base"`](#api); each item exposes `part="base"`,
`part="term"`, and `part="detail"`. Reach them with
[`::part()`](https://developer.mozilla.org/en-US/docs/Web/CSS/::part) for CSS
not covered by a token.

## Accessibility

- The description list is a semantic list: a `role="list"` wrapper with each
  pair as `role="listitem"`, so screen readers announce the count and iterate
  the pairs in reading order. Within a pair the term is read before its detail,
  mirroring how a native `<dl>` reads.
- A native `<dl>` is not used because the container must project authored items
  through a `<slot>`, which a shadow `<dl>` may not legally contain. The list /
  listitem role pairing produces an equivalent accessibility tree.
- Give the list an `aria-label` (or `aria-labelledby`) so it has an accessible
  name describing the subject the pairs are about.
- The component is presentational and non-interactive: there is no keyboard
  contract beyond the normal tab order of any interactive content you slot in
  (such as a link in a detail).
