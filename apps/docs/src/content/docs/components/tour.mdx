`<fluid-tour>` walks a user through your UI one step at a time. Give it an ordered
list of `steps` (each a `target` CSS selector plus a `title` and `body`) and it
spotlights the current target (a dimming scrim with a transparent cutout) and
shows a popover with a **Step n of m** counter and **Back / Next / Done** plus
**Skip** controls. ArrowLeft / ArrowRight move between steps, Escape skips, and
each step is announced to assistive tech. The scrim and popover render in the top
layer, so they are never clipped by an ancestor's overflow or transform.

## Examples

### Basic

`steps` is a property (an array of objects), so set it from script rather than as
an attribute. Each step needs a `target` selector, a `title`, and a `body`;
`placement` is optional.

```html
<fluid-tour id="onboarding"></fluid-tour>

<script type="module">
  const tour = document.getElementById("onboarding");
  tour.steps = [
    { target: "#search", title: "Search", body: "Find anything here.", placement: "bottom-start" },
    { target: "#new",    title: "Create", body: "Start a new project.", placement: "bottom" },
    { target: "#avatar", title: "Account", body: "Billing and settings.", placement: "left" }
  ];
  tour.show(); // or set the `open` attribute
</script>
```

### Controlling the tour

Drive it imperatively (`show()`, `next()`, `back()`, `finish()`, `skip()`) or
declaratively with the `open` and `index` attributes. Listen for the events to
persist progress or fire analytics.

```html
<fluid-tour id="t"></fluid-tour>

<script type="module">
  const t = document.getElementById("t");
  t.steps = [/* ... */];

  t.addEventListener("fluid-step-change", (e) => console.log("now on step", e.detail.index));
  t.addEventListener("fluid-finish", () => localStorage.setItem("tour-done", "1"));
  t.addEventListener("fluid-skip", () => localStorage.setItem("tour-skipped", "1"));

  if (!localStorage.getItem("tour-done")) t.show();
</script>
```

## When to use

- First-run onboarding: introduce the key parts of a new feature.
- Announcing a change: point at what moved after a redesign.
- Contextual coachmarks: a short, ordered walkthrough of a workflow.

Reach for a single [`<fluid-popover>`](/components/popover/) or
[`<fluid-tooltip>`](/components/tooltip/) when you only need one hint, not an
ordered, multi-step walkthrough.

## Theming

Every styled property reads a **component-scoped `--fluid-tour-*` token that
falls back to a main semantic var** (the override ladder): the popover surface,
the scrim color, the spotlight ring, and the primary button all retune
independently. The **complete list** is in the [API table below](#api).

```css title="brand.css"
[data-fluid-brand="custom"] {
  --fluid-tour-bg: var(--fluid-surface-base);
  --fluid-tour-scrim: rgb(0 0 0 / 0.6);
  --fluid-tour-highlight-ring: var(--fluid-accent-base);
  --fluid-tour-accent-bg: var(--fluid-accent-base);
  --fluid-tour-accent-fg: var(--fluid-accent-text);
}
```

### Beyond tokens: `::part()`

The popover is [`part="base"`](#api), with `part="scrim"`, `part="highlight"`,
`part="title"`, `part="body"`, `part="counter"`, `part="actions"`, and
`part="back"` / `part="next"` / `part="skip"` for the buttons. Reach them with
[`::part()`](https://developer.mozilla.org/en-US/docs/Web/CSS/::part) for CSS not
covered by a token.

## Accessibility

The tour popover follows the WAI-ARIA APG
[Dialog (Modal)](https://www.w3.org/WAI/ARIA/apg/patterns/dialog-modal/) pattern.

- The popover is `role="dialog"` with `aria-modal="true"`, labelled by its step
  title (`aria-labelledby`) and described by its body (`aria-describedby`).
- On open, focus moves into the popover and is trapped: Tab from the last control
  wraps to the first, Shift+Tab from the first wraps to the last.
- Each step is mirrored into a polite `aria-live` region ("Step n of m" plus the
  title and body) so a screen reader announces every change.
- Keyboard: <kbd>→</kbd> / <kbd>←</kbd> step forward / back, <kbd>Esc</kbd> skips
  the tour. When the tour closes, focus returns to whatever had it before.
- Buttons honor the conformance target-size token, so they grow to 44×44 under
  `data-fluid-conformance="aaa"`.
- Motion (the spotlight glide and the popover fade) honors
  `prefers-reduced-motion` and the `--fluid-motion` scalar.

### AA vs AAA

The component reads the conformance tokens rather than hard-coding sizes. Under
the default AA mode the buttons meet the 24×24 minimum target
([SC 2.5.8](https://www.w3.org/WAI/WCAG22/Understanding/target-size-minimum.html));
under `data-fluid-conformance="aaa"` `--fluid-target-min` lifts them to 44×44 and
`--fluid-focus-ring-width` / `--fluid-focus-ring-offset` widen the focus
indicator, with no per-component branching.
